2026-05-01: Implemented `yt items move <playlist-id> <item-id> --to <position>` (Milestone 1, item 2c) — completes Milestone 1 step 2.
  - New file: internal/cmd/items_move.go
  - Wired into items.go subcommand list
  - Args: <playlist-id> <item-id>; cobra.ExactArgs(2). Item id is the playlistItemId from `items list`, not videoId — long help spells this out.
  - Required flag: --to <int> zero-based target position. cmd.Flags().Changed("to") gates it (so `--to=0` is accepted but omitting --to errors). --to < 0 also rejected pre-API.
  - Quota: 1 (playlistItems.list to fetch current snippet/resourceId — required by API for update) + 50 (playlistItems.update) = 51 units. Documented in --help.
  - Safety check: after fetching the item, refuses to proceed if its snippet.PlaylistId doesn't match the playlist-id arg (catches typos / wrong-playlist mistakes before the 50-unit write).
  - Update payload: id + snippet{playlistId, resourceId, position}. ResourceId is copied from the fetched item — the API requires it on update or returns 400. Position=0 sets ForceSendFields=["Position"] so omitempty doesn't strip it (moving to top would otherwise no-op).
  - --dry-run prints "would move item X from position N to M in playlist P (cost: 51 units)" on stderr; the read still happens (1 unit) so we can validate ownership and surface the current position before the user commits to the 50-unit write
  - On success renders POS / ITEM_ID / VIDEO_ID / TITLE for the updated item; JSON output is the raw *PlaylistItem
  - Verified: go vet passes, build clean, `items move --help` documents 51-unit cost, arg-count + missing-`--to` + negative-`--to` errors all fire pre-API, command appears in `items` subcommand list
